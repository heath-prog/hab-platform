==================================================
CRITICAL PLATFORM UPGRADE: MULTI-DEAL / MULTI-BUSINESS ARCHITECTURE
==================================================

This platform must NOT be designed around a single acquisition.

The app must support:
- multiple leads
- multiple businesses under review
- multiple active acquisitions at the same time
- multiple closed shops under management
- future portfolio / roll-up ownership across multiple locations and entities

The user must be able to manage an entire acquisition pipeline and operating portfolio from one account.

Design the app as a portfolio-aware ownership operating system.

==================================================
PORTFOLIO STRUCTURE REQUIREMENT
==================================================

The product must distinguish between:

1. LEADS
Potential businesses not yet in diligence

2. DEALS
Businesses in active conversation / NDA / LOI / diligence / close workflows

3. ACQUIRED BUSINESSES
Closed shops that have moved into active operation and financial management

4. ARCHIVED / EXITED BUSINESSES
Disposed, closed, or inactive businesses retained for historical record

Each business should have its own dedicated workspace with:
- business name
- entity name
- deal stage
- acquisition type
- assigned team members
- documents
- AI verification results
- financials
- transition tasks
- reporting data
- historical timeline

==================================================
DOCUMENT PIPELINE ARCHITECTURE
==================================================

The document system must be multi-business aware.

When a document is uploaded in the app, it must automatically be associated with:
- portfolio owner
- business / deal
- document type
- document ID
- workflow stage
- upload timestamp
- uploader
- transaction path (stock / asset / both)
- original file source

Documents uploaded through the app are sent to an n8n workflow.

The workflow architecture must assume the following behavior:

1. ORIGINAL DOCUMENT STORAGE
- The uploaded file is stored in Google Drive
- It must use standardized naming conventions
- It must be organized for easy lookup by business and document type
- The original PDF/file must always remain retrievable from the app

2. STRUCTURED DATA EXTRACTION
- The n8n workflow extracts structured information from the document
- The extracted data is written into a data table, using Google Sheets as the practical v1 datastore
- The structured data is then available for accurate downstream workflow execution, reporting, analytics, and AI interpretation

3. VERIFICATION / ANALYSIS RETURN
- The workflow returns:
  - confidence score
  - summary
  - findings
  - red flags
  - recommendations
  - advisor notes
  - extracted data payload
  - drive file reference
  - document status indicators

4. ORIGINAL DOCUMENT RETRIEVAL
- The app must always allow the original source document to be retrieved from Google Drive as a PDF/file
- Users must be able to open or download the original document directly from the business workspace
- The structured data must never replace access to the source file

==================================================
GOOGLE DRIVE ORGANIZATION REQUIREMENT
==================================================

Design the file architecture so uploaded documents are stored in Google Drive in a clean and scalable way.

Recommended folder pattern:
- Portfolio
  - Business Name
    - Acquisition
    - Due Diligence
    - Legal
    - Lease
    - Financials
    - Licensing
    - Vendors
    - Post-Close
    - Reports

Use naming conventions such as:
{BusinessName} - {DocumentType} - {DocumentId} - {YYYY-MM-DD}

Examples:
True Blue Auto Care - Tax Returns - d1 - 2026-04-07.pdf
True Blue Auto Care - Lease Agreement - d2 - 2026-04-07.pdf

The UI should show both:
- human-friendly document name
- underlying source reference / file link

==================================================

The structured data extracted by the n8n workflow must be written into Google Sheets (or equivalent table abstraction in v1).

The app should treat this as a data table layer for:

Design the system so each extracted record is associated with:
- business ID

- document ID
- document type

- source file ID
- extracted fields
- verification confidence
- review status

The purpose of the data table is to allow more accurate calls and automations when the next workflow step needs to execute.

The system should use structured extracted data to power:
- checklist completion logic
- financial dashboards
- AI reporting
- compliance tracking
- due diligence summaries
- future investor/lender reports

==================================================
WORKFLOW EXECUTION LOGIC
==================================================

The product must be smart enough to use extracted structured data to trigger or support later workflows.

Example:
- If lease data has already been extracted into the table, lease workflow pages should read from that data
- If tax return revenue has already been extracted, financial dashboards and diligence pages should use that data
- If verification confidence is low, a manual review task should be triggered
- If a required document is missing for one business, only that business should show the blocker

The platform must behave like:
“The system remembers what was learned from every document and uses it later.”
